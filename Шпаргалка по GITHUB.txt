ШПАРГАЛКА ПО GIT – ВСЁ НА ОДНОМ ЛИСТЕ
=======================================

ЭТАП 0. ПЕРВЫЙ РАЗ НА НОВОМ КОМПЬЮТЕРЕ (один раз)
--------------------------------------------------
git config --global user.name "Ваше Имя"
git config --global user.email "почта@example.com"
# Зачем: Git подписывает коммиты этими данными. Без них не даст делать коммиты.

ЭТАП 1. НАЧАЛО ПРОЕКТА – ДВА ВАРИАНТА
---------------------------------------
ВАРИАНТ А: Создать новый репозиторий (у вас пока нет на GitHub, только локальная папка)
1. Зайдите в папку проекта (создайте её).
2. git init                      # включить Git в этой папке
3. git add .                     # добавить все файлы (следит за изменениями)
4. git commit -m "Первый коммит" # сохранить снимок
5. На сайте GitHub создайте НОВЫЙ ПУСТОЙ репозиторий (НЕ создавайте README, .gitignore, лицензию – будет пусто).
6. Скопируйте URL из поля "Quick setup" (https://github.com/ваш_логин/репо.git)
7. git remote add origin URL     # привязать локальную папку к удалённому репозиторию
8. git branch -M main            # переименовать ветку из master в main (чтобы совпадало с GitHub)
9. git push -u origin main       # отправить коммит на GitHub и запомнить связь

ВАРИАНТ Б: Склонировать существующий проект (уже есть на GitHub)
git clone https://github.com/чей-то_логин/репо.git
cd репо
# Всё! Уже настроено, можно сразу делать git add, git commit, git push.

ЭТАП 2. ПРО РЕПОЗИТОРИЙ С README НА GITHUB (важно!)
----------------------------------------------------
Если вы создали репозиторий на GitHub и поставили галку "Add a README file", то при первом push возникнет ошибка:
"failed to push some refs" – потому что на GitHub уже есть коммит (README), а у вас локально нет.

ЧТО ДЕЛАТЬ:
git pull origin main --allow-unrelated-histories   # объединить две независимые истории
# (возможно, откроется редактор для сообщения merge – просто закройте, нажав :wq в vim)
# Затем:
git push
# В следующий раз просто git pull (без --allow-unrelated-histories) и git push.

СОВЕТ: Если вы только начинаете проект, лучше НЕ создавать README на сайте – добавьте его позже через git add/commit/push.

ЭТАП 3. ЕЖЕДНЕВНАЯ РАБОТА (цикл)
---------------------------------
git add .            # или git add index.html (конкретный файл)
git commit -m "что сделал"
git push             # отправить на GitHub

Перед push полезно сделать git pull – подтянуть чужие изменения (если работаете в команде).

ЭТАП 4. ВЕТКИ (зачем? – чтобы экспериментировать, не ломая main)
-----------------------------------------------------------------
# Посмотреть все ветки (* – текущая)
git branch

# Создать новую ветку и переключиться на неё
git checkout -b новая_ветка
# или git switch -c новая_ветка

# Переключиться на существующую ветку
git checkout main
# или git switch main

# Слить ветку в текущую (например, вы в main, хотите добавить changes)
git merge новая_ветка

# Удалить ветку (локально) после слияния
git branch -d новая_ветка

# Удалить ветку на GitHub
git push origin --delete новая_ветка

ЭТАП 5. ОТКАТЫ – ВОЗВРАТ НАЗАД ВО ВРЕМЕНИ
------------------------------------------
# Сначала посмотрите историю (ID коммитов)
git log --oneline

# БЕЗОПАСНЫЙ ОТКАТ (создаёт новый коммит, не теряет историю)
git revert ID_коммита

# ОПАСНЫЙ ОТКАТ (стирает коммиты – используйте только если вы один работаете)
git reset --hard ID_коммита   # вернуться к старому состоянию + удалить всё новее
# Если перемудрили с --hard, не паникуйте:
git reflog                    # найти ID состояния, которое было до отката
git reset --hard найденный_ID

# Мягкий откат (убирает коммит, но оставляет изменения в файлах)
git reset --soft HEAD~1

# Смешанный откат (убирает коммит и изменения из индекса, но файлы сохраняются)
git reset HEAD~1

ЭТАП 6. РАЗОШЛИСЬ ПУТИ – ЧИНИМ СВЯЗЬ С GITHUB
---------------------------------------------
СИТУАЦИЯ 1: Вы переустановили систему, но папка с проектом осталась (есть .git).
- Проверьте, видит ли Git удалённый репозиторий: git remote -v
- Если пусто, добавьте: git remote add origin https://github.com/ваш_логин/репо.git
- Если URL старый и не работает: git remote set-url origin https://новый_адрес.git
- Затем git pull origin main (если нужно подтянуть), потом git push.

СИТУАЦИЯ 2: Вы на новом компьютере и хотите продолжить работу над проектом.
- Просто git clone https://github.com/ваш_логин/репо.git
- Внутри уже будет вся история и связь. Работайте как обычно.

СИТУАЦИЯ 3: Вы перенесли папку с одного ПК на другой, Git внутри есть (папка .git), но нужно просто продолжить.
- Откройте терминал в этой папке.
- git remote -v – если ссылка есть и правильная, ничего не делайте (просто git pull, git push).
- Если ссылки нет – добавьте (см. ситуацию 1).
- Если ссылка есть, но репозиторий переехал (например, сменили логин) – git remote set-url origin новый_URL.

ЭТАП 7. ЗАЧЕМ ПЕРЕИМЕНОВЫВАТЬ master В main?
---------------------------------------------
Git по умолчанию создаёт ветку master. GitHub с 2020 года рекомендует main.
Переименование нужно для синхронности: когда вы делаете git push -u origin main, на GitHub должна существовать ветка main.
Если вы оставите master, то на GitHub создадите ветку master – это тоже нормально, просто будьте последовательны.
Команда git branch -M main переименовывает текущую ветку (обычно master) в main – это один раз для проекта.

ЭТАП 8. ЧТО ДЕЛАТЬ, ЕСЛИ git push ОТКЛОНЯЕТ (rejected)
--------------------------------------------------------
Причина: на GitHub появились новые коммиты (вы или коллеги уже отправили что-то).
Решение:
git pull --rebase    # подтянуть чужие изменения и наложить свои поверх (чистая история)
# ИЛИ
git pull             # скачать и сделать merge (создаст дополнительный коммит слияния)
Затем снова git push.

КОРОТКИЙ СПИСОК ЧАСТО НУЖНЫХ КОМАНД
-----------------------------------
git status                # состояние: что изменено, что добавлено
git diff                  # показать изменения в строках
git log --oneline --graph # красивое дерево коммитов
git checkout -- файл      # отменить несохранённые изменения в файле
git reset HEAD файл       # убрать файл из индекса (отменить git add)

СИНХРОНИЗАЦИЯ ЛОКАЛЬНОЙ ВЕТКИ С УДАЛЁННОЙ
------------------------------------------------
Проблема: при переключении на ветку (например, main) Git сообщает:
  "Your branch is behind 'origin/main' by N commits"

Причина: на GitHub (удалённый репозиторий) появились новые коммиты, которых нет у вас локально.
Это происходит, например, после того, как вы слили Pull Request через веб-интерфейс или кто-то другой запушил изменения.

Решение (безопасное, сохраняет ваши локальные изменения, если они есть):
  git pull origin main

Если вы уверены, что локальных изменений нет, и хотите просто догнать удалённую ветку:
  git pull --ff-only origin main

Если вы случайно сделали локальные коммиты, которые ещё не отправили, и хотите их сохранить, а потом обновить:
  1. Создайте резервную ветку: git branch backup
  2. Затем git pull origin main
  3. После этого merge ваших изменений, если нужно.

Проверить состояние после синхронизации:
  git status

Примечание: "fast-forward" означает, что коммиты можно просто "накатить" без создания лишнего merge-коммита.

ИТОГ: Git хранит всё. Не бойтесь экспериментировать – всегда можно вернуться через reflog.
