ГЛАВНОЕ ПРАВИЛО (запомните! прочитайте внимательно)

# !!! ВАЖНОЕ ПРЕДУПРЕЖДЕНИЕ (чтобы не попасть впросак) !!!
# Если вы начнёте коммитить прямо в ветку main (или master), то:
# - Потом не сможете создать нормальный Pull Request (PR) для сдачи домашнего задания.
# - PR будет пустым, потому что main и так содержит все ваши изменения.
# - Преподаватель не увидит процесс вашей работы и не сможет комментировать код.
# - Вам придётся изворачиваться, пересоздавать ветки, пушить пустые коммиты – как бывает в таких случаях.

# ПРАВИЛЬНЫЙ ПОРЯДОК (запомните как молитву):
# 1. Всегда начинайте с создания новой ветки (`git checkout -b lesson1`).
# 2. Коммитьте только в эту ветку (пока локально).
# 3. По завершении работы – пуш ветки на GitHub.
# 4. На сайте GitHub создаёте Pull Request из ЭТОЙ ветки в main.
# 5. Копируете ссылку НА PULL REQUEST (в ней есть `/pull/1`) и отправляете преподавателю.
# 6. Только ПОСЛЕ проверки сливаете (или преподаватель сливает) ветку в main.

# ПРАВИЛЬНАЯ ССЫЛКА ДЛЯ СДАЧИ ДОМАШКИ:
# Это ссылка на страницу Pull Request, а не на репозиторий.
# Формат: https://github.com/ваш_логин/репозиторий/pull/1
# Часть `/pull/` есть всегда. Номер (1, 2, 3...) – какой по счёту PR вы создали.
# Без `/pull/` ссылка не принимается (это будет просто ссылка на репозиторий).

# !!! Про номера в Pull Request (чтобы не путаться) !!!
# Номер (1, 2, 3...) — это СКВОЗНОЙ счётчик GitHub для всего репозитория.
# Он общий для Issues, Pull Requests и Discussions.
# Номера никогда не переназначаются. Если в репозитории был PR #2, и его закрыли,
# следующему PR достанется #3, а не #2.
# Это не «номер ветки». Ссылка вида `/pull/5/pull/3` — НЕ СУЩЕСТВУЕТ.

# Никогда не коммитьте напрямую в main! Ветка – черновик, main – чистовик.

 – ПЕРВОНАЧАЛЬНАЯ НАСТРОЙКА И БАЗА

ЭТАП 0. НАСТРОЙКА GIT (один раз на компьютере)
----------------------------------------------------------------
# 1. Представьтесь (подписывает все ваши коммиты)
`git config --global user.name "Ваше Имя"`
`git config --global user.email "почта@example.com"`

# 2. (Опционально, т.е. по желанию) Настройте удобный редактор для длинных сообщений коммитов
#    По умолчанию Git использует сложный редактор Vim. Удобнее настроить свой.
#    Редактор понадобится, когда вы будете писать многострочные сообщения без флага -m.
#    Например, для VS Code:
`git config --global core.editor "code --wait"`
#    Для Notepad++ (Windows). Используйте двойные обратные слеши `\\`:
`git config --global core.editor "'C:\\Program Files\\Notepad++\\notepad++.exe' -w"`
#    Достаточно одного флага `-w` (wait) – Git будет ждать, пока вы закроете редактор.
#    Остальные флаги (`-multiInst`, `-notabbar`, `-nosession`, `-noPlugin`) – не обязательны.
#      - `-multiInst`	– открывать новый файл в новом окне, а не во вкладке.
#      - `-notabbar`	– скрыть полоску вкладок (просто визуально).
#      - `-nosession`	– не загружать старые файлы из сессии (ускоряет запуск).
#      - `-noPlugin`	– не загружать плагины (ускоряет запуск, если плагины тяжелые).
#    Если вам непонятны эти флаги – просто НЕ используйте их. Оставьте только `-w`.

# 3. Как проверить текущие настройки редактора:
`git config --global core.editor`
#    Если команда ничего не вывела (пустая строка) – значит, редактор не настроен.
#    Git будет использовать редактор по умолчанию (обычно Vim или блокнот).
#    Это нормально. Вы можете оставить как есть или настроить удобный редактор по желанию.

# 4. Как отменить настройку (вернуть стандартный редактор Git):
`git config --global --unset core.editor`

# 5. Как сделать сложный коммит с подробным описанием в редакторе:
`git commit`          # (без флага -m). Git откроет настроенный редактор.
#    - В первой строке напишите заголовок (до 50 символов).
#    - Оставьте пустую строку.
#    - Начиная с третьей строки, подробно опишите изменения.
#    Затем сохраните файл (Ctrl+S) и закройте редактор.
#    Git сам завершит коммит с вашим сообщением. Если передумали — закройте редактор без сохранения.

ЭТАП 1. НАЧАЛО ПРОЕКТА – ДВА ВАРИАНТА
----------------------------------------------------------------
ВАРИАНТ А: Создать новый репозиторий (у вас пока нет на GitHub, только локальная папка)

# --- ЛОКАЛЬНЫЕ ДЕЙСТВИЯ (всё происходит в папке .git на вашем компьютере) ---
1. Зайдите в папку проекта (создайте её).
2. `git init`						# инициализировать репозиторий (создать его в текущей папке).
3. `git add .`						# добавить все файлы из папки и подпапок в индекс (подготовить к коммиту).
									# проверить добавленное можно командой `git status`.
4. `git commit -m "Первый коммит"` 	# создать первый коммит (сохранить снимок состояния).

# --- ПОДКЛЮЧЕНИЕ К УДАЛЁННОМУ РЕПОЗИТОРИЮ НА GITHUB ---
5. На сайте GitHub создайте НОВЫЙ ПУСТОЙ репозиторий. Можно это сделать и до работы на локалке (НЕ создавайте README, .gitignore, лицензию – будет пусто).
6. Скопируйте URL из поля "Quick setup" (https://github.com/ваш_логин/репо.git) ← Ваш адрес будет похож на этот.
7. `git remote add origin <URL>`    # привязать локальную папку к удалённому репозиторию.
8. `git branch -M main`           	# (локально) переименовать ветку из master в main.
									# далее действия уже касаются сайта GitHub.
9. `git push -u origin main`      	# первый пуш (флаг -u запоминает связь, в дальнейшем можно просто `git push`).

ВАРИАНТ Б: Склонировать существующий проект (уже есть на GitHub) - тут есть два способа.

# --- Способ 1 (обычный, рекомендуемый для новичков) ---
# Git создаст НОВУЮ ПОДПАПКУ с именем репозитория в ТЕКУЩЕЙ папке.
# Например: `git clone https://github.com/чей-то_логин/репо.git`
#   появится папка `репо` – внутри код и скрытая папка `.git`.
`git clone https://github.com/чей-то_логин/репо.git`

# Затем перейдите в эту папку, чтобы работать:
`cd репо`

# --- Способ 2 (клонирование прямо в текущую папку) ---
# Если вы хотите, чтобы код репозитория оказался НЕ в подпапке, а прямо здесь,
# добавьте в конце команды точку:
`git clone https://github.com/чей-то_логин/репо.git .`

# ВАЖНО: так можно делать ТОЛЬКО в ПУСТОЙ папке!
# Например: Вы CMD находитесь в папке «Мои документы» (где полно других файлов и папок) и выполните:
`git clone https://github.com/чей-то_логин/репо.git .`
# Git попытается распаковать все файлы репозитория прямо в «Мои документы», вперемешку с вашими личными документами.
# Хорошая новость: Git не даст этого сделать, если целевая папка не пуста.
# Он выдаст ошибку: fatal: destination path '.' already exists and is not an empty directory.
# То есть клонирование с точкой требует абсолютно пустую текущую папку.
# Это сделано специально, чтобы вы случайно не засорили важные каталоги.

# Для новичков безопаснее использовать Способ 1 (без точки).

================================================================
ЭТАП 2. БАЗОВЫЕ ЕЖЕДНЕВНЫЕ КОММИТЫ (без веток)
----------------------------------------------------------------
# Самый короткий цикл:
`git add .`			# Можно добавить один или сколько указано файлов: `git add file-name.расширение`.
					# Указывайте файлы без запятой друг задругом через пробел.
`git commit -m "что сделал"`
`git push`			# Изменения ушли на сервер

# КОМАНДА `git commit -a -m "сообщение"` (с флагом -a) – в чём разница?
# Флаг -a (all) автоматически добавляет в коммит ВСЕ ИЗМЕНЁННЫЕ файлы, которые из-за изменений УЖЕ ОТСЛЕЖИВАЮТСЯ Git'ом.
# То есть вам не нужно писать `git add .` перед коммитом для таких файлов.
# НО:
#   - Новые файлы (которые вы только что создали и ещё ни разу не добавляли) НЕ будут включены.
#     Их нужно отдельно добавить: `git add новый_файл.py`
#   - Удалённые файлы (которые вы удалили из папки) НЕ будут автоматически удалены из Git.
#     Их нужно удалить командой `git rm имя_файла`.
# Пример:
`git commit -a -m "Сохранил изменения во всех существующих файлах"`

# Когда использовать `git commit -a -m`?
# - Когда вы правили только существующие файлы и хотите быстро закоммитить все изменения.
# - Когда не нужно выборочно исключать файлы из коммита.
# Когда использовать обычный `git commit -m`?
# - Когда вы добавили новые файлы (`git add`).
# - Когда хотите сделать коммит только части изменений.

# ВАЖНО: `git commit -a -m` НЕ добавляет новые файлы и НЕ обрабатывает удаления.
# Если вы создали новый файл, сначала: `git add файл`, потом `git commit` (с -m или без -a).
# `git commit -a -m` Работает только сизменёнными файлами , а не с созданными. И нужен для разового коммита по всем изменениям если их очень много и повсюду.

 – РАБОТА С ВЕТКАМИ

ЭТАП 3. ОСНОВЫ ВЕТОК
----------------------------------------------------------------
# В Git простая двухуровневая структура: есть основной ствол `main` (или `master`),
# и от него растут боковые ветки (например, `lesson1`). Ветки нужны, чтобы работать
# над разными задачами отдельно, не трогая основной код.

# Посмотреть все ветки (звёздочка * – текущая ветка)
`git branch`

# ВСЕГДА переключайтесь на main, затем создавайте новую ветку:
`git checkout main`
`git checkout -b новая_ветка`

# Переключиться на существующую ветку
`git checkout main`

# Слить изменения из другой ветки в текущую (например, вы в `main`, хотите добавить всё, что сделано в `lesson1`)
`git merge lesson1`

# Удалить ветку (локально) после слияния
`git branch -d новая_ветка`

# Удалить ветку на GitHub
`git push origin --delete новая_ветка`

# --- ТЕХНИЧЕСКОЕ ПОЯСНЕНИЕ (для понимания) ---
# Вы можете создать ветку от любой другой ветки, а не только от `main`.
# Например, находясь в `lesson1`, выполните `git checkout -b lesson1-fix`.
# Тогда `lesson1-fix` будет указывать на тот же коммит, что и `lesson1`.
# Дальнейшие коммиты в `lesson1-fix` создадут отдельную линию (как отросток от ветки).
# Визуально это выглядит как «ветка на ветке».
# Однако для простоты в учебных проектах рекомендуется создавать ветки только от `main`.
# Так вы избежите путаницы с несколькими уровнями слияний.


================================================================
 – ГЛАВНЫЙ ЦИКЛ ДЛЯ СДАЧИ ДОМАШНЕГО ЗАДАНИЯ
================================================================

ЭТАП 4. ПРАВИЛЬНЫЙ АЛГОРИТМ ДЛЯ КАЖДОЙ ДОМАШКИ
----------------------------------------------------------------
# Перед началом нового задания обновите main
`git checkout main`
`git pull origin main`

# Создайте новую ветку для задания (например, lesson1)
`git checkout -b lesson1`

# Теперь работайте: пишите код, создавайте файлы, делайте коммиты
`git add .`                     	# или `git add конкретный_файл`
`git commit -m "Сделано первое упражнение"`
# ... ещё несколько коммитов по ходу работы

# Когда задание полностью готово, отправьте ветку на GitHub
# (в первый раз для этой ветки обязательно с -u)
`git push -u origin lesson1`

# Затем на сайте GitHub создайте Pull Request (PR):
# - base: main, compare: lesson1
# - Не нажимайте "Merge" до проверки преподавателя
# - Скопируйте ссылку на PR (в адресной строке обязательно присутствует `/pull/1`, `/pull/2` и т.д.)

# Отправьте эту ссылку преподавателю — это и есть сдача домашнего задания.

# ПОСЛЕ того, как преподаватель одобрил и вы (или он) слили PR в main:
`git checkout main`
`git pull origin main`           	# обновить локальный main
`git branch -d lesson1`          	# удалить локальную ветку (она больше не нужна)
`git push origin --delete lesson1`  # удалить ветку на GitHub (по желанию)

# ЗАПОМНИТЕ: если вы закоммитили в main, нормальный PR сделать уже не получится.
# Всегда работайте в отдельной ветке.


================================================================
ЧТО ДЕЛАТЬ, ЕСЛИ ВЫ УЖЕ ЗАКОММИТИЛИ В main (а PR нужен)
----------------------------------------------------------------
Ситуация: вы сделали все коммиты в main (ваша история есть, но ветки нет).
Pull Request из main в main невозможен, а созданная позже ветка (копия main) даст пустой PR. (мост с нулевой длиной)

# РЕШЕНИЕ (без потери истории)
1. Найдите хэш коммита, который был ДО того, как вы начали делать упражнения.
   `git log --oneline`   # например, коммит со шпаргалкой (6a11bb3)

2. Создайте НОВУЮ ветку от этого старого коммита:
   `git checkout -b lesson1-clean <хэш>`
   # (замените <хэш> на найденный, например 6a11bb3)

3. Скопируйте ВСЕ ваши готовые файлы с упражнениями из текущей папки в новую ветку.
   # Физически файлы уже есть на диске. Просто добавьте их:
   `git add .`

4. Сделайте коммит (один или несколько, если хотите разбить историю):
   `git commit -m "Все упражнения урока 1"`

5. Отправьте ветку на GitHub:
   `git push -u origin lesson1-clean`

6. Создайте Pull Request из ветки `lesson1-clean` в `main`.
   Теперь PR покажет все добавленные файлы (изменения).

7. После проверки и слияния (Merge) удалите старую, ненужную ветку.

# Если вы не хотите возиться с поиском хэша, можно проще:
   Просто дайте преподавателю ссылку на репозиторий (не на PR).
   Он увидит историю коммитов во вкладке `Commits`. Это допустимо.

# ГЛАВНОЕ: в следующий раз начинайте с ветки ДО коммитов.

 – ПОЛЕЗНЫЕ КОМАНДЫ И РЕШЕНИЕ ПРОБЛЕМ

ЭТАП 5. ПЕРЕИМЕНОВАНИЕ ИЛИ ПЕРЕМЕЩЕНИЕ ФАЙЛОВ (git mv)
----------------------------------------------------------------
# Безопасный способ (рекомендуемый):
`git mv старое_имя.py новое_имя.py`
`git commit -m "Переименовал файл"`

# Что делает `git mv`?
# - переименовывает (перемещает) файл в файловой системе (как обычная команда mv)
# - автоматически добавляет изменение в индекс: старый файл помечается как удалённый,
#   новый – как добавленный.

# Если вы переименовали файл вручную (через mv или проводник):
`mv старое_имя.py новое_имя.py`
`git add новое_имя.py`   # Git начинает отслеживать новый файл
`git rm старое_имя.py`   # Git перестаёт отслеживать старый файл (физически он уже переименован)

# `git rm` здесь не удаляет файл с диска, а только убирает его из отслеживания.

================================================================
ЭТАП 6. ПРО РЕПОЗИТОРИЙ С README НА GITHUB (важно!)
----------------------------------------------------------------
Если при создании репозитория на GitHub вы поставили галку "Add a README file", то при первом push возникнет ошибка.

Решение:
`git pull origin main --allow-unrelated-histories`
# (если открылся редактор vim, нажмите :wq и Enter)
`git push`

# Лучше: создавайте репозиторий без README, а потом добавьте его сами через `git add`/`commit`/`push`.

================================================================
ЭТАП 7. ОТКАТЫ – ВОЗВРАТ НАЗАД ВО ВРЕМЕНИ
----------------------------------------------------------------
# Сначала посмотрите историю (ID коммитов)
`git log --oneline`

# БЕЗОПАСНЫЙ ОТКАТ (создаёт новый коммит, не теряет историю)
`git revert ID_коммита`

# ОПАСНЫЙ ОТКАТ (стирает коммиты – используйте только если вы один работаете)
`git reset --hard ID_коммита`
# Если перемудрили: `git reflog`, затем `git reset --hard найденный_ID`

# Мягкий откат (убирает коммит, но оставляет изменения в файлах)
`git reset --soft HEAD~1`

================================================================
ЭТАП 8. СИНХРОНИЗАЦИЯ (когда "Your branch is behind")
----------------------------------------------------------------
Проблема: `git status` говорит «Your branch is behind 'origin/main' by N commits».

Причина: на GitHub появились новые коммиты, которых нет у вас локально (например, после слияния PR).

Решение:
`git pull origin main`

Альтернатива с перебазированием (даёт чище историю):
`git pull --rebase origin main`

Затем снова `git push`.

================================================================
ЭТАП 9. ВОССТАНОВЛЕНИЕ СВЯЗИ С GITHUB
----------------------------------------------------------------
СИТУАЦИЯ 1: Папка с проектом есть, но remote потерялся.
- Проверьте: `git remote -v`
- Если пусто, добавьте: `git remote add origin https://github.com/ваш_логин/репо.git`

СИТУАЦИЯ 2: URL репозитория изменился.
`git remote set-url origin https://github.com/новый_логин/репо.git`

СИТУАЦИЯ 3: Новый компьютер – просто клонируйте.
`git clone https://github.com/ваш_логин/репо.git`

================================================================
ЭТАП 10. ЧТО ДЕЛАТЬ, ЕСЛИ git push ОТКЛОНЯЕТ (rejected)
----------------------------------------------------------------
Проблема: `git push` выдаёт ошибку "rejected non-fast-forward".

Причина: на GitHub появились новые коммиты (например, кто-то запушил или вы через веб-интерфейс).

Решение:
`git pull`
# (или `git pull --rebase`)

Затем снова:
`git push`

================================================================
ЧАСТЫЕ ОШИБКИ НОВИЧКОВ И ИХ РЕШЕНИЕ
----------------------------------------------------------------
1. «Я всё закоммитил в main, а потом создал ветку lesson1 и открыл PR – он пустой».
   Ошибка: работа должна делаться В ВЕТКЕ, а не в main.
   Исправление: начинайте с `git checkout -b новая_ветка` ДО того, как писать код.

2. «Не могу создать PR – нет кнопки "Compare & pull request"».
   Причина: ветка не отправлена на GitHub.
   Решение: `git push -u origin ваша_ветка`.

3. «PR показывает "no changes"».
   Причина: вы пытаетесь сравнить две одинаковые ветки (main и её копию).
   Решение: работайте в отдельной ветке с момента начала.

4. «git push rejected (non-fast-forward)».
   Решение: `git pull`, затем `git push`.

5. «Команда `git commit -a -m` не добавила в коммит новый файл!»

   Команда `git commit -a -m` — это способ быстро закоммитить все изменения в уже отслеживаемых файлах.
   - Флаг -a (от англ. all) автоматически добавляет в коммит изменения во всех файлах, которые Git уже знает.
   - Флаг -m (от англ. message) позволяет указать сообщение прямо в строке.

   ВАЖНО: Эта команда НЕ включает в коммит новые файлы, созданные вами в последний раз. Для новых файлов сначала обязательно используйте `git add <имя_файла>`.

   Что команда делает на самом деле (простыми словами):
   1. Находит все файлы, которые Git уже отслеживает, и замечает в них изменения.
   2. Автоматически их подготавливает к коммиту (делает `git add` за вас).
   3. Сразу же создаёт коммит с вашим сообщением.

   Пример:
   - Вы изменили существующий файл `index.html`.
   - Вы создали новый файл `style.css`.
   - Выполнив `git commit -a -m "Обновил стили"`, вы:
     * + добавите в коммит изменения в `index.html`;
     * - НЕ добавите новый файл `style.css`.
   - Итог: ваш новый файл останется неотслеживаемым. Правильно будет сначала сделать `git add style.css`, а потом `git commit -m "..."`.

   Подытожим: `git commit -a -m` помогает, когда вы меняете существующий код, но не подходит, когда вы добавляете совершенно новые файлы.

================================================================
КОРОТКИЙ СПИСОК КОМАНД (всегда под рукой)
----------------------------------------------------------------
`git status`                # что изменилось
`git diff`                  # показать изменения в строках
`git log --oneline --graph` # красивое дерево коммитов
`git checkout -- файл`      # отменить несохранённые изменения в файле (обязательно два тире и пробел)
`git reset HEAD файл`       # убрать файл из индекса (отменить git add)

================================================================
ИТОГ (КРАТКИЙ АЛГОРИТМ ДЛЯ КАЖДОЙ ДОМАШКИ)
----------------------------------------------------------------
1. Никогда не коммитьте в main. main — это чистовик для принятого кода.

2. Перед началом работы обновите main и создайте новую ветку:
   `git checkout main`
   `git pull origin main`
   `git checkout -b lesson1`          # ветка для задания

3. Работайте в этой ветке. Делайте коммиты:
   `git add .`
   `git commit -m "Сделал то-то"`

4. Когда задание готово, отправьте ветку на GitHub:
   `git push -u origin lesson1`       # первый пуш ветки

5. На сайте GitHub создайте Pull Request (PR):
   - `base: main`, `compare: lesson1`
   - Не нажимайте "Merge" до проверки преподавателя.
   - Скопируйте ссылку на PR (в адресной строке обязательно присутствует `/pull/1`, `/pull/2` и т.д.).

6. Отправьте эту ссылку преподавателю — это и есть сдача домашнего задания.

7. После одобрения нажмите "Merge pull request" на GitHub.

8. Обновите локальный main и удалите отработанную ветку:
   `git checkout main`
   `git pull origin main`
   `git branch -d lesson1`
   `git push origin --delete lesson1`   # удалить ветку на GitHub (по желанию)

ЗАПОМНИТЕ: коммиты напрямую в main — это ошибка. Всегда работайте в отдельных ветках.
